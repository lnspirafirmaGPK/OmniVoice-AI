# AGENTS.md

## ภาพรวมโปรเจกต์

**OmniVoice AI** คือแพลตฟอร์ม **AI รับสายโทรศัพท์อัตโนมัติสำหรับธุรกิจ E-commerce** ที่ทำหน้าที่เสมือน **พนักงานขายอัจฉริยะ** สามารถรับสาย ตอบคำถามลูกค้า แนะนำสินค้า ปิดการขาย คัดกรองลีด ยืนยันคำสั่งซื้อ และนัดหมายได้ **ตลอด 24 ชั่วโมง** โดยเชื่อมต่อกับระบบร้านค้าและเครื่องมือหลังบ้านของธุรกิจ

เอกสารนี้ใช้เป็นแนวทางสำหรับผู้พัฒนา มนุษย์ และ AI agents ที่เข้ามาทำงานใน repository นี้ เพื่อให้ทุกการแก้ไขโค้ด เอกสาร และสถาปัตยกรรม สอดคล้องกับวิสัยทัศน์ของโปรเจกต์

> หมายเหตุ: หากพบชื่อเดิมเช่น `OmniCore AI` ในเอกสารเก่าบางส่วน ให้ถือว่าเป็น legacy naming และให้ใช้ชื่อ **OmniVoice AI** เป็นชื่อหลักของผลิตภัณฑ์สำหรับงานใหม่ เว้นแต่กำลังอ้างอิงข้อความเดิมโดยตรง

## เป้าหมายหลักของระบบ

ระบบนี้ต้องช่วยให้ธุรกิจ E-commerce สามารถ:

1. **รับสายอัตโนมัติ 24/7** โดยไม่พลาดโอกาสการขาย
2. **ตอบคำถามลูกค้าได้อย่างแม่นยำ** จากข้อมูลสินค้า โปรโมชัน คำถามพบบ่อย และเงื่อนไขร้านค้า
3. **ปิดการขายผ่านเสียงสนทนา** ด้วยบทสนทนาที่เป็นธรรมชาติ สุภาพ และมุ่งผลลัพธ์
4. **ดึงข้อมูลจากระบบร้านค้า** เช่น สต็อก ราคา คำสั่งซื้อ สถานะจัดส่ง และข้อมูลลูกค้า
5. **สร้างนัดหมายหรือส่งต่อเจ้าหน้าที่** เมื่อเคสซับซ้อนหรือมีมูลค่าสูง
6. **บันทึกและสรุปผลการสนทนา** เพื่อใช้ติดตามผล วัด KPI และปรับปรุงคุณภาพการขาย
7. **ทำงานหลายช่องทางได้ในอนาคต** เช่น โทรศัพท์, LINE, web chat, CRM, และ dashboard ภายใน

## สิ่งที่ AI agents ต้องยึดถือเมื่อทำงานกับ repo นี้

### 1) ยึด product vision ก่อน implementation
ทุกการเปลี่ยนแปลงต้องตอบคำถามได้ว่า:
- ช่วยให้ระบบรับสายและขายได้ดีขึ้นหรือไม่
- ช่วยให้ธุรกิจใช้งานจริงได้ง่ายขึ้นหรือไม่
- ลดภาระพนักงาน เพิ่ม conversion หรือเพิ่มคุณภาพบริการหรือไม่

### 2) Thai-first และ business-first
ค่าเริ่มต้นของประสบการณ์ใช้งานควรเน้น:
- ภาษาไทยเป็นหลัก
- ใช้งานง่ายสำหรับเจ้าของธุรกิจทั่วไป
- ลดศัพท์เทคนิคในหน้าใช้งาน
- รองรับบริบทการขายจริงของธุรกิจไทย

### 3) เชื่อมโลก “เสียงสนทนา” กับ “ข้อมูลร้านค้า”
ฟีเจอร์ทุกตัวควรถูกออกแบบให้เชื่อมกันระหว่าง:
- การโทรเข้า / โทรออก
- ข้อมูลสินค้าและราคา
- คำสั่งซื้อและการชำระเงิน
- การนัดหมายและการติดตามผล
- CRM / lead history / customer profile

### 4) Human handoff ต้องมีเสมอ
ระบบต้องสามารถส่งต่อคนจริงได้เมื่อ:
- ลูกค้าต้องการคุยกับเจ้าหน้าที่
- AI ไม่มั่นใจในคำตอบ
- เคสเกี่ยวข้องกับการร้องเรียน ปัญหาสำคัญ หรือความเสี่ยงสูง
- เคสมูลค่าสูงที่ควรให้ฝ่ายขายดูแลต่อ

## บทบาทของ Agents ภายในระบบ

AI agents ที่พัฒนาภายใต้โปรเจกต์นี้ควรถูกออกแบบให้แยกบทบาทชัดเจน เพื่อให้ทดสอบและขยายระบบได้ง่าย

### 1. Reception Agent
หน้าที่:
- รับสาย ทักทาย ตรวจจับเจตนาเบื้องต้น
- ตรวจว่าลูกค้าต้องการถามสินค้า ติดตามออเดอร์ นัดหมาย หรือคุยกับพนักงาน
- route ไปยัง agent หรือ workflow ที่เหมาะสม

### 2. Sales Agent
หน้าที่:
- แนะนำสินค้าโดยอ้างอิงจากข้อมูลจริงในระบบ
- ตอบข้อสงสัยเกี่ยวกับราคา โปรโมชัน คุณสมบัติ และความแตกต่างของสินค้า
- ใช้เทคนิคการขายอย่างสุภาพ เช่น upsell, cross-sell, bundle suggestion
- ปิดการขายหรือเก็บความสนใจของลูกค้าให้อยู่ใน funnel

### 3. Order Support Agent
หน้าที่:
- ตรวจสอบสถานะคำสั่งซื้อ
- ยืนยันข้อมูลลูกค้า/เบอร์โทร/เลขออเดอร์
- ให้ข้อมูลการจัดส่ง การชำระเงิน และขั้นตอนหลังการขาย

### 4. Appointment Agent
หน้าที่:
- นัดหมายเวลาที่พนักงานจะโทรกลับ
- จองคิวเดโม ปรึกษา หรือ follow-up
- บันทึกวัน เวลา ช่องทางติดต่อ และเหตุผลของการนัดหมายให้ครบถ้วน

### 5. Escalation Agent / Human Handoff Layer
หน้าที่:
- ส่งต่อไปยังทีมขายหรือแอดมิน
- สรุปบริบทการสนทนาก่อนส่งต่อ
- ลดการให้ลูกค้าต้องพูดข้อมูลซ้ำ

### 6. QA / Analytics Agent
หน้าที่:
- สรุปผลการสนทนา
- จัดหมวดหมู่ disposition
- วัดคุณภาพบทสนทนา conversion และปัญหาที่เกิดซ้ำ
- ช่วยหา insight เพื่อปรับปรุง script และฐานความรู้

## หลักการออกแบบบทสนทนา

ทุก agent ที่เกี่ยวข้องกับลูกค้าต้องยึดหลักต่อไปนี้:

- **สุภาพ ชัดเจน กระชับ**
- **ขายได้ แต่ไม่กดดันเกินไป**
- **พูดเหมือนพนักงานขายมืออาชีพ ไม่ใช่หุ่นยนต์**
- **ยืนยันข้อมูลสำคัญก่อนทำ action** เช่น เบอร์โทร ชื่อสินค้า จำนวน นัดหมาย
- **ไม่เดา** เมื่อข้อมูลไม่พอ
- **อ้างอิงเฉพาะข้อมูลจากระบบที่เชื่อถือได้**
- **เมื่อไม่แน่ใจให้เสนอทางเลือก** เช่น ให้เจ้าหน้าที่ติดต่อกลับ
- **ลดระยะเวลาสนทนาโดยไม่เสียคุณภาพ**

### แนวทางภาษาที่ควรใช้
- ใช้ภาษาไทยธรรมชาติ เข้าใจง่าย
- หลีกเลี่ยงคำอังกฤษหรือศัพท์เทคนิคหากไม่จำเป็น
- สำหรับลูกค้าใหม่ ควรเน้นประโยชน์ของสินค้าและความมั่นใจในการสั่งซื้อ
- สำหรับลูกค้าเดิม ควรเน้นความต่อเนื่อง เช่น ประวัติการซื้อ สถานะออเดอร์ และข้อเสนอเฉพาะบุคคล

## แหล่งข้อมูลที่ระบบควรเชื่อมต่อ

ระบบควรออกแบบให้เชื่อมต่อหรือรองรับการเชื่อมต่อกับ:

- ระบบร้านค้า: Shopify, WooCommerce, custom e-commerce backend
- CRM / lead database
- ระบบโทรศัพท์ / SIP / telephony provider
- Calendar / appointment service
- Chat / LINE / email / notification system
- Inventory / order / shipping providers
- Knowledge base / vector database / FAQ store
- Analytics / logging / monitoring stack

เมื่อพัฒนา integration ใหม่ ให้ยึดหลัก:
- แยก adapter ตาม provider
- มี interface กลางที่เปลี่ยน provider ได้
- ไม่ผูก business logic เข้ากับ vendor เดียว

## แนวทางสถาปัตยกรรม

หากยังไม่มีโครงสร้างตายตัว ให้ยึดแนวคิดนี้เป็น baseline:

### ชั้นสำคัญของระบบ
1. **Telephony Layer**
   - รับสาย / วางสาย / โอนสาย / callback / recording event
2. **Speech Layer**
   - speech-to-text, text-to-speech, interruption handling, latency control
3. **Agent Orchestration Layer**
   - intent detection, tool calling, workflow routing, memory, guardrails
4. **Commerce Integration Layer**
   - products, prices, carts, orders, shipping, customer data
5. **Knowledge Layer**
   - FAQ, policy, scripts, product knowledge, embeddings / retrieval
6. **Business Action Layer**
   - create lead, create appointment, update CRM, notify staff, log disposition
7. **Dashboard / Admin Layer**
   - configuration, monitoring, transcript review, analytics, prompt management

### สถาปัตยกรรมที่ควรหลีกเลี่ยง
- business logic กระจุกอยู่ใน controller เดียว
- prompt ยาวทั้งหมด hardcode ไว้หลายจุด
- ไม่มี typed schema สำหรับ tool input/output
- ผูก voice logic, sales logic, และ integration logic ไว้ในไฟล์เดียว

## มาตรฐานการเขียนโค้ด

### หลักทั่วไป
- เขียนโค้ดให้ **อ่านง่ายก่อนฉลาด**
- แยก concern ให้ชัด
- เพิ่ม type definitions / schema validation เมื่อเหมาะสม
- หลีกเลี่ยง magic string และค่าคงที่กระจัดกระจาย
- ทุก flow สำคัญต้องมี logging ที่พอใช้ debug ได้

### Naming
- ใช้ชื่อที่สื่อความหมายทางธุรกิจ เช่น `salesAgent`, `appointmentBooking`, `orderLookup`
- หลีกเลี่ยงชื่อกว้างเกินไป เช่น `helper`, `utils2`, `tempFlow`
- ตั้งชื่อ event/action ตามพฤติกรรมจริง เช่น `CALL_STARTED`, `LEAD_CREATED`, `HUMAN_HANDOFF_REQUESTED`

### Configuration
- ค่า environment ต้องรวมอยู่ใน `.env.example`
- แยก config ตาม environment
- ห้าม hardcode secrets, tokens, private URLs, API keys

### Error handling
- error ที่ลูกค้าเห็นต้องอ่านรู้เรื่อง
- error ภายในต้องมี context พอสำหรับ trace
- external provider call ต้องมี timeout, retry policy, และ fallback ที่เหมาะสม

## แนวทางสำหรับ AI/LLM workflow

เมื่อ AI agent แก้โค้ดหรือเพิ่มฟีเจอร์ที่เกี่ยวกับ LLM ให้ยึดแนวทางนี้:

### Prompt design
- แยก prompt ออกจาก business logic เท่าที่ทำได้
- เก็บ prompt versioning ได้
- ระบุ objective, tone, constraints, และ allowed actions ชัดเจน
- ใส่ fallback behavior เมื่อข้อมูลไม่พอ

### Tool calling
- ทุก tool ต้องมี schema ชัดเจน
- ตรวจสอบ input ก่อนยิงไปยัง external system
- tool ที่มีผลต่อข้อมูลจริง เช่น สร้างนัดหมายหรือแก้คำสั่งซื้อ ต้องมี confirmation step

### Retrieval / knowledge
- ใช้เฉพาะข้อมูลที่เชื่อถือได้
- แยก knowledge ของสินค้า นโยบายร้าน และ script การขายออกจากกัน
- ควรมีวิธีอัปเดตฐานความรู้โดยไม่ต้องแก้โค้ดหลักบ่อย

### Guardrails
- ห้าม fabricate ราคา สต็อก ส่วนลด หรือนโยบายร้าน
- ห้ามรับปากแทนมนุษย์ในเรื่องที่ระบบยังยืนยันไม่ได้
- ห้ามให้คำแนะนำเฉพาะทางที่เกินขอบเขตธุรกิจ

## ความปลอดภัยและความเป็นส่วนตัว

โปรเจกต์นี้เกี่ยวข้องกับข้อมูลลูกค้าและการสื่อสารผ่านโทรศัพท์ จึงต้องระวังเป็นพิเศษ

### ต้องทำ
- ปกป้องข้อมูลส่วนบุคคล เช่น ชื่อ เบอร์โทร ที่อยู่ ประวัติการสั่งซื้อ
- จำกัดสิทธิ์การเข้าถึงข้อมูลตามบทบาท
- log เท่าที่จำเป็น และหลีกเลี่ยงการเก็บข้อมูลอ่อนไหวเกินจำเป็น
- mask ข้อมูลสำคัญใน log และ analytics
- ชี้แจงเรื่องการบันทึกเสียงหรือ transcript ตามนโยบายที่ใช้งานจริง

### ห้ามทำ
- commit secrets ลง repo
- เก็บ PII ในไฟล์ทดสอบแบบไม่ปกปิด
- เปิด endpoint ภายในโดยไม่มี auth หรือ rate limiting ที่เหมาะสม

## แนวทางการทดสอบ

ทุกการเปลี่ยนแปลงที่กระทบ flow ธุรกิจ ควรมีการทดสอบอย่างน้อยหนึ่งแบบจากด้านล่าง:

- unit tests สำหรับ business logic
- integration tests สำหรับ external APIs / store connectors / telephony adapters
- conversation tests สำหรับ intent และ sales flow สำคัญ
- regression tests สำหรับ FAQ, product recommendation, appointment booking, order lookup

### เส้นทางสำคัญที่ต้องไม่พัง
- รับสายแล้วตอบได้
- ถามข้อมูลสินค้าแล้วดึงคำตอบจากฐานข้อมูลได้
- ตรวจสอบออเดอร์ได้
- สร้างลีดได้
- สร้างนัดหมายได้
- ส่งต่อมนุษย์ได้
- สรุปผลหลังจบสายได้

## Definition of Done

งานหนึ่งชิ้นถือว่าเสร็จเมื่อ:

- สอดคล้องกับ product vision ของ OmniVoice AI
- ไม่ทำลาย flow รับสาย/ขาย/นัดหมายเดิม
- มี error handling ที่เหมาะสม
- มี logging หรือ observability ที่พอใช้งาน
- มีการอัปเดตเอกสารที่เกี่ยวข้อง
- ชื่อฟีเจอร์และข้อความในระบบสื่อสารชัดเจนกับผู้ใช้ธุรกิจ

เมื่อ AI agent แก้ README, docs, landing content, dashboard copy หรือ prompt templates:

- ใช้ชื่อ **OmniVoice AI** เป็นหลัก

- เน้นผลลัพธ์ทางธุรกิจ เช่น รับสายไม่พลาด, เพิ่มยอดขาย, ลดภาระทีม, นัดหมายอัตโนมัติ

- ถ้าต้องกล่าวถึงอนาคต ให้แยกจากสิ่งที่มีอยู่แล้วให้ชัดเจน

## ตัวอย่างข้อความอธิบายผลิตภัณฑ์มาตรฐาน

ใช้ข้อความนี้เป็น baseline ได้เมื่อต้องเขียนเอกสาร/หน้าเว็บ/คำอธิบาย repo:

> OmniVoice AI คือแพลตฟอร์ม AI รับสายโทรศัพท์อัตโนมัติสำหรับธุรกิจ E-commerce ที่เชื่อมต่อกับระบบร้านค้าเพื่อทำหน้าที่เป็นพนักงานขายอัจฉริยะ ตอบคำถามลูกค้า แนะนำสินค้า ปิดการขาย ตรวจสอบคำสั่งซื้อ และนัดหมายได้ตลอด 24 ชั่วโมง

ข้อความขยาย:

> ระบบช่วยให้ธุรกิจไม่พลาดทุกสายสำคัญ แม้นอกเวลาทำการ โดย AI สามารถสนทนาอย่างเป็นธรรมชาติ ดึงข้อมูลสินค้าหรือออเดอร์จากระบบร้านค้าแบบเรียลไทม์ สร้างลีด นัดหมาย หรือส่งต่อเจ้าหน้าที่ได้ทันที พร้อมบันทึกสรุปบทสนทนาและข้อมูลเชิงวิเคราะห์เพื่อช่วยเพิ่ม conversion และคุณภาพการบริการ

## ลำดับความสำคัญในการพัฒนา

เมื่อไม่แน่ใจว่าจะพัฒนาอะไรก่อน ให้เรียงตามนี้:

1. ความเสถียรของการรับสายและบทสนทนา
2. ความถูกต้องของข้อมูลสินค้า/ออเดอร์/นัดหมาย
3. ความสามารถในการปิดการขายและเก็บลีด
4. Human handoff และเครื่องมือให้ทีมขายทำงานต่อ
5. Dashboard และ analytics
6. ฟีเจอร์เสริมอื่น ๆ
